# VLM 엣지 캡처 해석 — 모델 선정 & 프롬프트 튜닝 기록 (Qwen3-VL-2B)

엣지(YOLO)가 캡처한 장면을 2B VLM이 **한국어로 얼마나 정확·상세히 해석**하는지 평가·튜닝한 기록 (종합 유형 정확도 **71% → 89%**).

## 데이터

- **이미지** — 34장
- **영상** — 36장
- **상황** — 화재 · 연기 · 낙상 · 전도 (위험) / 정상
- **환경** — RTX 4060 (8GB) · transformers 5.9.0 · fp16

---

## 모델 선정 (`Qwen2-VL-2B` vs `Qwen3-VL-2B`)
**결론 — `Qwen3-VL-2B-Instruct` 채택.**   
- Qwen2-VL-2B 는 **한국어 지시를 거부**("이미지를 이해할 수 없다")하지만 **영어로는 정상 동작**한다.   
- 차이는 순수하게 **모델의 한국어 멀티모달 지시-수행 능력**이다.   
- **입력 파이프라인은 정상** — 비전 토큰 삽입 · `pixel_values`(정상 범위·NaN 없음) · `image_pad` 패치 정합 · `image_grid_thw` · fp16/bf16 동일까지 확인. → 거부는 **모델의 응답 행동** 문제.

### Qwen2-VL-2B-Instruct — 한국어 거부, 영어 정상
- 동일 모델·동일 이미지인데 한국어는 전부 거부, 영어는 정확히 해석.

| 프롬프트 | 출력(요약) |
|---|---|
| 🇰🇷 "이 이미지에 무엇이 보이나요? 자세히 설명해줘." | ❌ "죄송합니다, 제가 현재 이 문제에 대해 이해할 수 없습니다." |
| 🇰🇷 "이 CCTV 캡처에 화재·연기·쓰러짐 같은 위험이 있나요?" | ❌ "죄송합니다, 제가 이 정보를 제공할 수 없습니다." |
| 🇰🇷 구조화 프롬프트(판정/유형/설명) | ❌ "죄송합니다, 하지만 현재 저는 **이미지나 영상에 대한 이해 능력이 없습니다.** …" |
| 🇺🇸 "What do you see in this image? Describe it in detail." | ✅ "The image appears to be a blurred photograph … a **news report about an 'accused Ontario warehouse arsonist in court tomorrow'** … **NBC 4** …" |
| 🇺🇸 영어 지시 + 구조화(낙상 이미지) | ✅ "VERDICT: danger / TYPE: fall / … people gathered around a **stretcher** with a person lying on it … emergency responders … the situation is serious" |



### Qwen3-VL-2B-Instruct — 한국어·영어 모두 정상
-  Qwen2-VL(2024) 대비 Qwen3-VL(2025-10)은 다국어·한국어 정렬이 크게 개선됨 (2B 소형에서 특히 두드러짐)   
-  Qwen2-VL은 영어로는 멀쩡하므로 "성능 부족"이 아니라 **한국어 지시 수행 실패**
-  **참고** — 비교 이미지는 유튜브 검색 기반 대용 데이터라 노이즈 있음 (예: "화재" 첫 이미지가 실제 화재가 아니라 "방화범 재판" 뉴스 화면)

| 프롬프트 | 출력(요약) |
|---|---|
| 🇺🇸 영어 지시 | ✅ "VERDICT: danger / TYPE: fall / [한국어] 사진은 밤에 대형 사고 현장에서 … 병원용 침대에 누워 있는 사람을 안전하게 이동 …" |
| 🇰🇷 구조화 프롬프트(판정/유형/설명) | ✅ "**판정: 위험 / 유형: 낙상 / 설명:** 이 이미지에는 밤에 야외에서 인근 병원·구조대 장비를 사용한 구조 작업이 진행 … 환자를 안정적으로 운반 …" |



## 모델 정교화

### 1) 다중분류 오분류 해결 (판정 형식)

- **문제**: 판정 형식을 `판정: 위험 또는 정상` 으로 제공 → `위험 또는 정상`을 그대로 출력(둘 다 선택).
- **결과**: 형식을 `판정: 위험 / 정상` 으로 변경 + 프롬프트에 *"위험·정상 중 하나만 적으세요"* 명시.

---

### 2) 프롬프트 개선 (P1 → P5) — 설명 간결화

| 시도 | 핵심 시도 | 이미지 정확도 | 영상 정확도 | 오탐 | 위험(이미지/영상) | 정상(이미지/영상) |
|---|---|---|---|---|---|---|
| **P1** | 구조화(판정/유형/설명) + *"가능한 한 자세히"* | **94%** | 83% | 0% | 353 / 359 | 360 / 347 |
| **P2** | few-shot 예시 1개 추가 | 85% | 78% | 10% | 233 / 243 | 248 / 254 |
| **P3** | JSON 형식 강제 출력 | 76% (미상 5) | 69% | 10% | 170 / 154 | 162 / 157 |
| **P4** | P3 설명 문구 차용 + *"자세히" 제거* | 91% | 83% | 0% | 184 / 248 | 184 / 226 |
| **P5** | P4 + *"정상이면 위험 나열 말고 결론만"* | **94%** | 83% | 10% | **111 / 117** | **122 / 114** |

#### 프롬프트별 판정 vs 유형 정확도 (이미지 / 영상)

|  | 판정 정확도 | 유형 정확도 |
|---|---|---|
| P1 | 94 / 83 | 76 / 75 |
| P2 | 85 / 78 | 65 / 64 |
| P3 | 76 / 69 | 74 / 72 |
| P4 | 91 / 83 | 71 / 75 |
| **P5** | **94 / 83** | 71 / 72 |

#### P5 카테고리별 판정 vs 유형 정확도 (이미지 / 영상)

| 카테고리 | 판정 정확도 | 유형 정확도 |
|---|---|---|
| 화재 | 100 / 100 | 83 / 100 |
| 연기 | 86 / 89 | 71 / 78 |
| 사람 쓰러짐/낙상 | 100 / 60 | 100 / 60 |
| **기계·장비 전도** | 100 / 60 | **0 / 0** |
| 정상 주차장 | 100 / 100 | 100 / 100 |
| 정상 작업장 | 80 / 75 | 80 / 75 |

→ 기계 전도는 판정('위험')은 맞히지만 **유형을 전부 '낙상'으로 오인(0%)**.

#### 시도별 개선 포인트

- **P1 → P2 (few-shot)**: 출력은 짧아졌으나 정확도·오탐 악화. few-shot 예시가 오히려 판정을 흐림.
- **P3 (JSON)**: 가장 짧지만 **정확도 최저 + 미상(분류 못하는 케이스) 발생** → 2B 소형 모델은 JSON 강제에 취약. 부적합.
- **P4 (핵심 발견)**: P1의 **"가능한 한 자세히" 지시가 장황함의 주원인**. 빼니 길이 절반(353→184자), 정확도는 91%로 거의 유지.
- **P5 (정상 장면)**: 정상 장면에서 *"화재 없음·낙상 없음·전도 없음…"* 식으로 모든 위험을 부정하며 장황해지던 문제를 **"나열 말고 결론만"** 규칙으로 해결 → 정상 길이 184→122자, 전체 111자로 최단이면서 정확도 94% 회복.

#### 인사이트

1. **"자세히" 지시 = 장황함의 주범** — 빼면 간결해지고 정확도는 유지됨.
2. **JSON 강제는 2B 모델에 부적합** — 정확도↓·미파싱↑.
3. **정상 장면은 위험 '부재'를 나열하며 길어짐** — "결론만" 규칙으로 통제.
4. **중국어(한자) 혼입은 프롬프트로 못 막음** — 생성 단계 토큰 차단(FORCE_KOREAN)으로 해결.
5. 판정/형식 에코는 **형식 단순화 + 채점 엄격화** 양쪽으로 잡아야 함.

#### 결론

- **채택: P5** (또는 정확도-우선 시 P1) — 가장 간결(111자)하면서 정확도 94% 유지.
- **공통 미해결 약점**: 영상의 낙상·전도(40~60%)는 프롬프트로는 한계 → 입력 파라미터 튜닝 필요(아래 3).
- 반복 루프(같은 문장 반복) → `repetition_penalty`·`no_repeat_ngram_size` 로 후속 조치 예정.

### 3) 영상 파라미터 튜닝 (프레임·해상도) — 전도 '판정' 개선

- **문제**: 기계 전도(움직임 위험)를 '위험'으로 잘 판정하지 못함(프레임이 적어 동작을 못 봄).
- **결과**: 영상의 프레임 수·해상도를 올려 개선. 특히 **기계 전도 판정이 프레임 수에 정비례**.
  - `4f 40% → 8f 60% → 16f 80% → 16f+560px 100%`
- **이미지는 개선 안 됨** — 수집 이미지가 이미 작아 픽셀 상한이 적용되지 않음(다운스케일 X).

| 파라미터 | 설명 |
|---|---|
| `VIDEO_NUM_FRAMES` | 영상을 몇 컷으로 쪼개 볼지 |
| `VIDEO_FRAME_MAX_SIDE` | 영상 프레임 품질(해상도) |
| `IMAGE_MAX_PIXELS` / `IMAGE_MIN_PIXELS` | 이미지 품질(해상도) |

| 프레임 | 변(px) | 판정 | 유형 | 지연 | VRAM |
|---|---|---|---|---|---|
| 4 | 448 | 75% | 67% | 2.5s | 4242MB |
| 8 | 448 | 83% | 72% | 2.5s | 4403MB |
| 8 | 640 | 86% | 72% | 3.0s | 4570MB |
| 16 | 448 | 86% | 72% | 2.8s | 4855MB |
| **16** | **560** | **89%** | **75%** | 3.5s | 5176MB |

→ **16프레임 + 560px 채택**(VRAM 5.2GB, 8GB 내).

### 4) 전도를 '낙상'으로 오분류 (유형)

- **문제**: 기계 전도를 구분하지 못하고 모두 **낙상으로 분류**(유형 0%). '전도(顚倒)'는 동음이의가 많아 단어를 바꿔도 안 됨.
- **결과**: 모델이 기계·사람 모두 "낙상"으로 분류하는 경향을 **역이용** → `낙상(기계)` / `낙상(사람)`으로 구분하도록 프롬프트 변경.
  - → 전도·낙상 분류율 **동반 상승**

| 유형 종류 | 전도 | 낙상 |
|---|---|---|
| 낙상 / 전도 | 0% | 80% |
| 낙상 / 전복 | 9% | 80% |
| 낙상 / 전도(기계·차량 넘어짐) | 0% | 80% |
| **낙상(사람) / 낙상(기계)** | **91%** | **100%** |

### 5) 화재를 '낙상'으로 오분류 (연기 회귀)

- **문제**: 유형을 `화재 / 낙상(사람) / 낙상(기계) / 없음`으로 바꾸며 **'낙상'이 2번** 등장 → 모델의 낙상 선택 확률↑.
  - → **연기 + 사람이 같이 찍힌 장면**이 화재가 아니라 낙상으로 분류.
- **결과**: 프롬프트에 우선순위 규칙 추가.

  > "연기·불꽃·화염이 보이면 사람이 함께 있어도 유형은 반드시 '화재'로 정하세요."

- 이전 정확도 확보 + **전도 분류 개선(91% → 100%)**.

|  | 화재 | 연기 | 낙상 | 전도 |
|---|---|---|---|---|
| 낙상(기계) | 93% | 62% | 100% | 91% |
| **+ 규칙 추가** | **100%** | **81%** | **100%** | **100%** |

### 6) 같은 말 반복 문제 — 프롬프트로 이미 해결 (전용 파라미터는 역효과)

- **문제**: 옛 프롬프트에선 일부 출력이 같은 문장을 수십 회 반복(예: fire7 영상 641자·25회).
- **검증**: `repetition_penalty` · `no_repeat_ngram_size` 를 올려 비교(반복 항목 + 정상 대조군).

| 설정 | 반복 | 평균 길이 | 형식 |
|---|---|---|---|
| base (1.05 / off) | 없음 | 짧음(63~176자) | 정상 |
| 1.15 / ngram3 | 없음 | 길어짐 | 정상 |
| 1.3 / ngram3 | 없음 | 더 길어짐(person_fall 142→284자) | 일부 유형 파싱 실패 |

- **결과**: 현재 프롬프트(firepri) 출력 **70건 전부 반복 0**(평균 119자) — 반복은 이미 P5 간결화 규칙이 해결.
  - 파라미터를 올리면 **고칠 반복이 없는데 출력만 길어지고**(간결화 역행) **구조 형식까지 깨짐** → **미채택**.
  - 결론: `repetition_penalty=1.05` 유지, `no_repeat_ngram_size=0`(off). (`REPEAT_TEST.md`)

### 7) 한자(중국어) 혼입 차단 — FORCE_KOREAN

- **문제**: 한국어로 지시해도 모델이 중국어 한자를 섞음(`分析`, `出現`처럼 한글 대신 한자 출력). **프롬프트 지시로는 못 막음**.

**적용 방식: 디코딩 후처리 (Logits Processing)**
- 모델의 파라미터를 수정한 것이 **아님**. 텍스트 생성 단계(`generate`)에서 제공되는 `logits_processor` 훅(Hook)을 활용.
- **동작 원리**: 모델이 다음 토큰(글자)을 예측할 때, 한자에 해당하는 토큰들(26,076개)의 확률(logit 점수)에 강제로 **−∞(마이너스 무한대) 페널티**를 부여해 생성 결과물에서 원천적으로 배제.
- `FORCE_KOREAN` 은 이 후처리 로직을 활성화하는 **커스텀 플래그**.

**검증 (on/off A/B, 23건)**

| | 한자 출현 | 총 한자 |
|---|---|---|
| **OFF** | 6/23건 | **39자** (`为像内出分因…`, `不切前危回接…`, `堆放` 등) |
| **ON** | **0/23건** | **0자** |

→ 끄면 26% 출력에 중국어가 섞이고, 켜면 완전 차단. 전체 운영 출력 946건 중 한자는 **로직 확정 전 최초 런 1건뿐**. (`FORCE_KOREAN_TEST.md`)

## 최종 결과 — 종합 유형 정확도 71% → 89%

| 카테고리 | 이미지 판정/유형 | 영상 판정/유형 |
|---|---|---|
| 화재 | 100 / 100% | 100 / 100% |
| 연기 | 86 / 57% | 100 / 100% |
| 낙상 | 100 / 100% | 60 / 100% |
| 전도 | 100 / 100% | 80 / 100% |
| 정상(주차/작업) | 100·80 / 80·60% | 100·75 / 80·75% |
| **종합 유형** | **82%** | **94%** |

- 전도 완전 해결(0→100%), 연기 영상 100%·이미지 회복, 어느 유형도 회귀 없음
- **남은 한계**: 이미지 연기 3건(불 미발생·전도오인·누운사람)은 2B 모델의 지각 한계 → few-shot/파인튜닝 영역
